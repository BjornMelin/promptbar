# Release Policy

Promptbar uses Release Please to automate release pull requests, SemVer version
updates, changelog entries, GitHub Releases, and Git tags.

## Source Of Truth

- `.github/workflows/release-please.yml` runs Release Please on pushes to
  `main` and on manual dispatch.
- `release-please-config.json` owns Release Please package strategy, changelog
  sections, pre-1.0 version mapping, and tag formatting.
- `bootstrap-sha` in `release-please-config.json` pins the seeded `0.1.0`
  baseline to the commit that was current on `main` before release automation
  was added.
- `.release-please-manifest.json` records the last released version for the root
  package.
- `package.json` stores the package version Release Please updates.
- `CHANGELOG.md` is generated and maintained by Release Please.
- `.github/pull_request_template.md` captures reviewer-facing release impact,
  validation, risk, and rollback details for normal PRs.
- `AGENTS.md` points automation agents to this policy and preserves repo-level
  invariants.

## Workflow

1. Feature and fix PRs merge to `main` with Conventional Commit titles.
2. The `Release Please` workflow runs on the `main` push.
3. Release Please opens or updates a release PR.
4. The release PR updates `package.json`, `.release-please-manifest.json`, and
   `CHANGELOG.md`.
5. Merging the release PR creates the GitHub Release and the matching Git tag.

## Authentication And Permissions

The workflow uses `secrets.RELEASE_PLEASE_TOKEN` so release PRs and release
events are authored by the configured maintainer credential instead of the
default `GITHUB_TOKEN`. This token must be a personal access token or GitHub App
token, not only for authorship but also because events and pull requests created
with `GITHUB_TOKEN` do not trigger subsequent workflow runs or CI. Keep Release
Please authenticated with `secrets.RELEASE_PLEASE_TOKEN` so generated release
PRs and release events can start the repository's normal CI and automation.

The workflow declares the Release Please write permissions it needs:

- `contents: write`
- `issues: write`
- `pull-requests: write`

The repository also has `RELEASE_COMMIT_NAME` and `RELEASE_COMMIT_EMAIL`
variables configured for release identity documentation. Release Please action
authorship is controlled by `RELEASE_PLEASE_TOKEN`.

## SemVer Track

Promptbar starts at `0.1.0` and remains on a pre-1.0 development track until
the repo explicitly moves to `1.0.0`.

Default pre-1.0 mapping:

- `fix:` creates a patch release, such as `v0.1.1`.
- `feat:` creates a patch release, such as `v0.1.1`.
- `!` or `BREAKING CHANGE:` creates a minor release, such as `v0.2.0`.

Mark breaking changes honestly with `!` or a `BREAKING CHANGE:` footer. Do not
use `release-as` unless a maintainer explicitly requests a one-off override.

## Conventional Commits

Pull request titles and squash merge commits should use:

```text
<type>(<optional-scope>): <concise imperative summary>
```

Common types:

- `feat`
- `fix`
- `refactor`
- `perf`
- `docs`
- `test`
- `build`
- `ci`
- `chore`
- `revert`

## Tags And Releases

Release tags must be plain SemVer tags with a leading `v`: `v0.1.0`, `v0.1.1`,
`v0.2.0`, and so on.

`release-please-config.json` sets:

- `include-component-in-tag: false`
- `include-v-in-tag: true`
- `include-v-in-release-name: true`

Do not introduce component-prefixed tags such as `promptbar-v0.1.1`.

## Changelog And Release Notes

Release Please owns generated changelog entries and GitHub Release notes. Keep
generated version headings parseable in release PR bodies and `CHANGELOG.md`;
do not replace them with custom headings that remove the version marker.

Visible changelog sections include features, bug fixes, performance,
documentation, dependencies, and reverts. Maintenance-only sections such as
refactors, tests, build, CI, and chores are hidden from the generated changelog
unless the release configuration changes.

## Review Requirements

Every non-release PR should fill out `.github/pull_request_template.md`,
including:

- expected Release Please bump;
- public API impact;
- breaking-change and migration details, when relevant;
- validation commands;
- risk and rollback notes.

Release PRs generated by Release Please should preserve the generated release
notes and version heading. Review those PRs for the intended version bump,
changelog content, `package.json`, `.release-please-manifest.json`, GitHub
Release title, and `vX.Y.Z` tag behavior.
